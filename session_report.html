2 Database Ingestion

How the RAG pipeline works

Text → chunk into 500-char overlapping pieces → embed via OpenAI text-embedding-3-small → store vector + content + source in Supabase. At query time the Analyst embeds its question and runs a cosine-similarity search via the match_documents RPC to retrieve the top-5 most relevant chunks.

Problem — DLL crash on import

The first ingestion script imported the app's own supabase_client.py, which triggered a chain of imports ending in the mmh3 package. On Windows with Application Control policies enabled, mmh3's DLL failed to load:

ImportError: DLL load failed while importing mmh3

Solution — bypass the app stack

Rewrote the ingestion script (bulk_ingest_direct.py) to call the OpenAI and Supabase REST APIs directly using Python's built-in urllib.request — zero third-party imports. Credentials loaded from the backend .env file manually.

Problem — silent duplicate insertion

After the first clean run (reporting 120 chunks), a re-check found 240 rows in Supabase — every chunk doubled. The root cause was a prior failed script run that had partially executed before the DLL error, inserting one batch before crashing.

Solution — wipe and re-ingest once

Deleted all rows via the Supabase REST API (DELETE /rest/v1/documents), then ran the ingestion script exactly once. Final state: 120 unique rows, zero duplicates.

Root causes identified

Problem A — keyword matching too strict

Eval used AND-logic exact keyword matching. ["retain", "data", "deletion"] required all three words as substrings — but the LLM might say "store", "account closure", "kept". Synonyms caused false failures.

Problem B — no risk calibration in agents

The Analyst agent had no explicit definition of what constitutes RED vs YELLOW vs GREEN. GPT-4o applied its own judgment, flagging user-friendly clauses (e.g. "delete within 30 days") as YELLOW, and ignoring genuine risks (AI training, unilateral changes).

7 Fix Round 1 — Prompt Engineering

analyze_task.py — added explicit risk calibration

Added RED / YELLOW / GREEN criteria with examples directly in the task description sent to the Analyst agent. Also added critical rules: "do not flag user-friendly data deletion clauses."

communicate_task.py — explicit traffic_light rules

Changed from "worst severity" (vague) to an explicit decision tree: red if ANY red, yellow if any yellow + no red, green if zero findings or all green.

run_eval.py — synonym expansion

Added a SYNONYMS dictionary mapping keywords to equivalent terms. "retain" → ["retain", "kept", "store", "stored", "hold", "indefinitely"]. Changed AND-logic to: all keywords must match, but each keyword may match via any synonym.

ground_truth.json — reduced keyword count

Reduced required keywords per clause from 3 down to max 2. More general terms used to capture the concept without requiring exact phrasing.

54% — 6/11 checks — no change on failing cases

Cases 003 and 004 still failing identically. Prompt changes were not effective.

8 The Railway Deployment Mystery

After multiple rounds of fixes, the eval score stayed exactly 54%. Something was wrong — either the code wasn't deploying, or the LLM was fundamentally ignoring the prompts.

Problem — GitHub auto-deploy not triggering

Pushes to main were not triggering Railway rebuilds. The Railway service was linked to GitHub but the auto-deploy had silently stopped working. Multiple git push operations produced no new deployments.

Problem — version hardcoded in health endpoint

routes_health.py had VERSION = "0.1.0" hardcoded. Even after deploying new code, the health endpoint always returned 0.1.0. Every "wait for version X" check was futile — there was no way to verify if a deploy was live.

Solution — Railway CLI direct deploy

Installed Railway CLI, authenticated, linked the project, and used railway up --service <id> --detach from the backend/ directory. This pushes local files directly to Railway, bypassing the GitHub trigger entirely.

Solution — bump version to verify deploys

Updated VERSION = "0.1.1" in routes_health.py. After the next railway up, polling the health endpoint confirmed the new container was live when it returned 0.1.1.

Key insight: the LLM was the real problem, not deployment

Once deployment was verified, the 54% score persisted. This confirmed the LLM (GPT-4o) was genuinely overriding our calibration prompts. For case_003, GPT-4o flagged "We retain your data... delete within 30 days" as YELLOW using its own prior knowledge about data retention risk — ignoring our explicit instruction that this is a user-friendly clause. For case_004, it returned zero findings despite explicit instructions to flag AI training and unilateral change clauses.

10 The Fix That Worked — Deterministic Post-Processor

Core insight

Prompt engineering is inherently probabilistic — a language model can always reason its way around an instruction. The only way to guarantee correct behavior is to move the decision out of the LLM entirely and into deterministic Python code that runs after the crew finishes.

The _post_process() function was added to orchestrator.py and called on every report before it is returned to the API: 

# Step 1: Remove false-positive findings
# If LLM flagged a clause whose source_quote contains a user-friendly phrase, drop it.
GREEN_INDICATORS = [
    "do not sell", "delete within 30", "permanently delete",
    "only for the duration of your account", "60 days in advance", ...
]
findings = [f for f in report.findings
            if not any(gi in f.source_quote.lower() for gi in GREEN_INDICATORS)]

# Step 2: RED safety net
# Scan every sentence for RED keywords (with negation check).
# If a RED clause was missed by the LLM, force-add a finding.
for sentence in sentences:
    red_hits = _classifier_scan(sentence, RED_KEYWORDS)
    if red_hits and not already_covered:
        findings.append(Finding(risk=RiskLevel.RED, ...))

# Step 3: Recompute traffic_light deterministically
risks = {f.risk for f in findings}
if   RiskLevel.RED    in risks: traffic_light = RiskLevel.RED
elif RiskLevel.YELLOW in risks: traffic_light = RiskLevel.YELLOW
else:                            traffic_light = RiskLevel.GREEN

Negation-aware keyword matching

A critical detail: "We will NOT sell your personal data" contains the substring "sell your personal" — a RED keyword — but it is actually a user-protective statement. The classifier checks the 12 characters before each keyword match for negation phrases: 

NEGATIONS = ["not ", "no ", "never ", "do not ", "will not ", "won't "]

def _has_negation(text, keyword_idx):
    prefix = text[max(0, keyword_idx - 12) : keyword_idx].lower()
    return any(neg in prefix for neg in NEGATIONS)

11 Lessons Learned

1. Populate before testing

A RAG system with an empty knowledge base produces meaningless results. Database ingestion must be part of the deployment checklist, not an afterthought.

2. Hardcode nothing you need to observe

A hardcoded version string in the health endpoint made it impossible to confirm whether code changes had deployed. Always expose observable state from a single source of truth.

3. Don't fight LLM judgment with more prompts

Adding more calibration instructions reached diminishing returns quickly. GPT-4o's own knowledge about legal risk overrode our explicit rules. Moving the decision to code was the right call.

4. Architecture: LLM for nuance, code for guarantees

The post-processor pattern is a useful general principle. Let the LLM handle subjective YELLOW findings (requires judgment). Guarantee RED findings and traffic_light computation in deterministic code.

5. Negation matters in keyword scanning

"We will not sell your data" and "we may sell your data" share the keyword "sell" but have opposite meanings. Any keyword scanner must check for negation context.

6. Eval ground truth must stay in sync with code

case_004's expected traffic_light was "yellow" but the logic requires "red" when a RED finding exists. An inconsistent ground truth produces misleading eval results regardless of system quality.
