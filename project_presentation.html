💡What Is TermsGuard AI?

Every time we click "I Agree" on a Terms of Service or Privacy Policy we hand over rights we never read. TermsGuard AI solves this with a multi-agent RAG pipeline that classifies legal clauses, looks them up against a knowledge base of 100+ real-world ToS documents, and delivers an instant risk report with exact citations back to the original text.

🔴 RED — High risk clause detected
🟡 YELLOW — Moderate risk, read carefully
🟢 GREEN — Looks standard / user-friendly

Core Problem Solved

Average ToS is 7,000 words and takes 30+ minutes to read. Legal language is deliberately opaque. Non-lawyers have no practical way to assess risk. TermsGuard bridges that gap with AI.

Key Features

🔍 Instant Analysis

Paste any ToS text and get a full risk report with traffic-light rating in ~10 seconds.

📌 In-Page Highlighting

Extension highlights the exact clauses in the original page — click a finding, the text lights up.

📚 Knowledge Base RAG

161 ingested documents including GDPR, CCPA, and ToS from 100+ companies like Google, Discord, OpenAI.

🤖 Multi-Agent Design

CrewAI orchestration with specialist agents for extraction, analysis, and plain-language communication.

🌐 Dual Interface

Chrome/Edge browser extension for live pages + Streamlit web UI for manual paste analysis.

☁️ Production Deployed

Live backend on Railway with GitHub Actions CI/CD pipeline for auto-deploy on push to main.

🤖Agents & Crew Orchestration

The project uses CrewAI 0.80 for multi-agent orchestration. Each agent has a defined role, backstory, and toolset, allowing the LLM to stay in-character and produce consistent outputs.

The Three Agents

📄 Extractor Agent

Role: Legal Document Structural Extractor
LLM: gpt-4o-mini (speed-optimised)
Purpose: Parse a ToS/Privacy Policy into discrete clauses with exact character offsets for downstream highlighting.
extractor.py

⚖️ Analyst Agent

Role: Privacy Lawyer & Risk Analyst
LLM: gpt-4o (full reasoning)
Purpose: Classify each clause as RED / YELLOW / GREEN using legal expertise + RAG knowledge base. This is the core intelligence.
analyst.py primary

✍️ Communicator Agent

Role: Plain-English Risk Communicator
LLM: gpt-4o-mini
Purpose: Transform analyst findings into jargon-free explanations a non-lawyer can understand.
communicator.py

Orchestration Pipeline

📥Inputraw_text + url
📚RAG FetchTop-4 KB chunks
✂️Truncate10,000 chars
🤖AnalystCrewAI + prompt
🔍Post-processFilter false+
📊ReportJSON response

Analyst Prompt Strategy

The agent prompt injects several layers of context:

You are an expert privacy lawyer with 20 years GDPR/CCPA experience.

[LEGAL KNOWLEDGE BASE]
{rag_context}          ← top-4 similar chunks from 161-doc KB

[DOCUMENT TO ANALYZE]
{truncated_tos_text}   ← up to 10,000 chars

Classification rules:
  RED    → selling data, arbitration waiver, indefinite retention,
           perpetual IP licence, unilateral modifications
  YELLOW → targeted ads, data sharing, AI training, auto-renew,
           <14 days notice, 2+ year retention
  GREEN  → explicit no-sell, 60+ day notice, data export rights,
           delete on request

Output ONE JSON object: { traffic_light, summary, findings[] }

Post-Processing Filter

After the agent responds, a deterministic filter removes false positives — findings with phrases like "do not sell", "permanently delete", or "60 days notice" are automatically downgraded. This prevents the LLM from flagging user-friendly clauses as risky.

🗄️Database — Supabase + pgvector

The database layer is Supabase (managed PostgreSQL) with the pgvector extension for high-dimensional vector similarity search. No separate vector DB is needed — everything lives in one place.

Schema

-- Migration: 20260423073802_remote_commit.sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE documents (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  content    TEXT        NOT NULL,           -- chunk text (≤1500 chars)
  embedding  vector(1536) NOT NULL,          -- OpenAI text-embedding-3-small
  source     TEXT,                           -- source name or URL
  metadata   JSONB,                          -- { "name": "...", "category": "..." }
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Semantic similarity search RPC
CREATE OR REPLACE FUNCTION match_documents(
  query_embedding vector(1536),
  match_count     INT DEFAULT 5
)
RETURNS TABLE (id UUID, content TEXT, source TEXT, metadata JSONB, similarity FLOAT)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT id, content, source, metadata,
         1 - (documents.embedding <=> query_embedding) AS similarity
  FROM documents
  ORDER BY documents.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;

Why Supabase?

One service, two jobs

Acts as both relational DB and vector store. No need to manage Pinecone, Weaviate, or Chroma separately.

Managed hosting

Free tier sufficient for development. No infra overhead — just connect with URL + service role key.

pgvector cosine search

Native SQL operator <=> for cosine distance on 1536-dimensional embeddings.

JSONB metadata

Each chunk stores arbitrary metadata (source name, category) alongside embedding — no extra table needed.

Supabase Client

# services/supabase_client.py — singleton pattern

🧩Frontend — Chrome/Edge Extension

The extension is a Manifest V3 Chrome extension that works on any ToS/Privacy Policy page. It auto-detects legal documents, extracts the text, and calls the backend.

How It Works — Step by Step

Step 1 — Page Load
Content Script Detects ToS Page
content.js runs on every page. It scans the URL for patterns like /privacy, /terms, /legal and looks for "I agree" buttons. If detected, it sets the extension badge to "ToS" in yellow.
Step 2 — User Click
Popup Opens, User Hits "Analyze"
Popup injects a content script via chrome.scripting.executeScript to extract the full page text. A 3-step progress animation plays (~4s extracting → ~25s analyzing → preparing).
Step 3 — API Call
Text Sent to Backend
api_client.js POSTs to https://backend-production-1d50.up.railway.app/api/v1/analyze with the raw text, URL, and page title.
Step 4 — Results Rendered
Traffic Light + Finding Cards
Each finding is rendered as a colored card (red/yellow/green). The popup shows finding count, total latency, and source URL.
Step 5 — Highlighting
Click Card → Text Highlights on Page
highlighter.js uses progressive substring matching: tries the full source_quote, then 120 chars, 80 chars, 50 chars. Uses window.find() + <mark> DOM injection with risk-color backgrounds. Scrolls into view automatically.

📈Development Evolution

The project went through significant architectural changes from initial plan to final version. Here is the honest story of what was planned, what changed, and why.

Phase 1 — Foundation (Planned vs. Reality)

Planned: Simple single-agent ToS analyzer, basic FastAPI endpoint, minimal frontend.

What happened: Phase 1 expanded significantly — full CrewAI multi-agent skeleton (Extractor → Analyst → Communicator), Supabase RAG layer, and even CI/CD pipelines were all scaffolded from day one. Commits show the full architecture being laid down: models, agents, tools, routes, tests, Docker, GitHub Actions — all in the first week.

• Replaced Communicator with direct LLM call (bypassed CrewAI overhead)
• Tried job queue pattern to bypass timeout (worked but complex)
• Increased truncation from 8000 → 15000 chars (reverted — slower)
• Final solution: single-agent single-call pipeline — all three roles merged into one agent + one prompt. Result: ~8s consistently.
